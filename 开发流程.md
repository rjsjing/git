# 待办事项应用开发全流程

## 第一步：项目初始化

- 在本地桌面创建 `git` 文件夹
- 初始化 Git 仓库（`git init`）
- 在 GitHub 创建远程仓库并关联本地

## 第二步：配置 AI 编程环境

- 安装 Claude Code
- 接入 DeepSeek API
- 配置 Git 用户信息

## 第三步：搭建 Node.js 项目骨架

- 创建 `package.json`，声明启动命令和依赖
- 创建 `.gitignore`，排除 `node_modules/`
- 执行 `npm install` 安装 Express

## 第四步：编写后端 REST API

- 创建 `server.js`，定义 5 个 API 端点：
  - GET /api/todos（查询，支持 ?q= 搜索）
  - GET /api/todos/:id（获取单个）
  - POST /api/todos（新增）
  - PUT /api/todos/:id（更新）
  - DELETE /api/todos/:id（删除）
- 数据暂存内存数组

## 第五步：编写前端界面

- 创建 `public/todo.html`，包含 HTML+CSS+JS
- 实现添加、勾选完成/未完成、删除、搜索功能
- 所有数据操作通过 `fetch` 调用后端 API

## 第六步：前后端对接

- 在 `server.js` 中添加 `app.use(express.static('public', { index: 'todo.html' }))`
- 修改前端，将 `localStorage` 替换为 API 调用
- 启动后访问 `http://localhost:3000` 直接展示待办页面

## 第七步：数据持久化

- 引入 `fs` 和 `path` 模块
- 创建 `readData()` 函数，从 `data.json` 读取数据（文件不存在返回空数组）
- 创建 `writeData()` 函数，将数据写入 `data.json`（JSON 格式化，缩进 2 空格）
- 服务器启动时从文件加载数据，自动计算 `nextId`
- 所有增删改操作后自动调用 `writeData()` 同步到文件
- 在 `.gitignore` 中添加 `data.json`，避免用户数据提交到仓库
- 重启服务器后数据不丢失

## 第八步：部署上线（Railway）

- 创建 `Procfile`，声明启动命令 `web: node server.js`
- 推送代码到 GitHub
- 在 Railway 中用 GitHub 登录，选择仓库自动部署
- Railway 自动检测 `package.json` 中的 `start` 脚本和 `Procfile`
- 生成公网地址，例如 `https://git-production-xxx.up.railway.app`
- 在 Railway Settings → Environment Variables 中配置 `TODO_PASSWORD`

## 第九步：分类标签

**后端：**
- todo 数据结构增加 `tag` 字段，默认为 "其他"
- POST /api/todos 接收 `{ text, tag, dueDate }`，tag 为空时默认 "其他"
- PUT /api/todos/:id 支持更新 tag
- GET /api/todos 新增 `?tag=工作` 按标签筛选
- 新增 GET /api/tags 路由，返回所有去重标签列表

**前端：**
- 输入框和添加按钮之间增加标签下拉菜单（工作/个人/学习/其他）
- 列表中每个事项显示彩色标签 pill：
  - 工作 = 蓝色、个人 = 绿色、学习 = 紫色、其他 = 灰色
- 搜索框和列表之间增加标签筛选按钮组（全部 + 四个标签）
- 点击标签按钮筛选对应分类，当前选中按钮高亮

## 第十步：截止日期

**后端：**
- todo 数据结构增加 `dueDate` 字段（可选，格式 YYYY-MM-DD）
- POST /api/todos 接收 `{ text, tag, dueDate }`
- PUT /api/todos/:id 支持更新 dueDate
- GET /api/todos 新增 `?overdue=true` 返回已过期且未完成的事项
- 列表按截止日期自动排序：有日期在前（越近越前），无日期在后

**前端：**
- 工具栏增加日期选择器（`<input type="date">`）
- 每个事项显示截止日期（格式：月/日，如 12/25）
- 日期颜色规则：
  - 已过期且未完成 → 红色加粗
  - 今天到期 → 橙色
  - 未来 → 正常灰色
  - 已完成 → 正常灰色

## 第十一步：用户登录保护

**后端：**
- 引入 Node.js 内置 `crypto` 模块
- 从环境变量读取密码：`TODO_PASSWORD`（默认 `admin123`）
- 新增 POST /api/login 接口：
  - 验证密码，正确返回 `{ token: base64(密码) }`，错误返回 401
- 创建 `authMiddleware` 中间件：
  - 从 `Authorization: Bearer <token>` 提取 token
  - base64 解码后与密码比对，不匹配返回 401
- 所有 `/api/todos` 路由受 `app.use('/api/todos', authMiddleware)` 保护
- GET /api/tags 同样需要认证
- 静态文件（public）和登录接口（/api/login）不受拦截

**路由注册顺序（关键）：**
```
app.use(express.json())                    // 公开
app.use(express.static('public'))          // 公开
app.post('/api/login', ...)                // 公开
app.use('/api/todos', authMiddleware)      // 认证墙
app.get('/api/todos', ...)                 // 以下全部需要认证
app.get('/api/tags', authMiddleware, ...)
app.get('/api/todos/:id', ...)
app.post('/api/todos', ...)
app.put('/api/todos/:id', ...)
app.delete('/api/todos/:id', ...)
```

**前端：**
- 页面加载时检查 `localStorage` 中是否有 token：
  - 无 token → 显示登录界面
  - 有 token → 显示主界面
- 登录界面：居中卡片 + 密码输入框 + 登录按钮 + 错误提示
- 封装 `api(method, path, body)` 函数，自动附加 `Authorization` 头
- API 返回 401 时自动清除 token 并跳回登录页
- 主界面右上角"退出登录"按钮，点击清除 token

## 涉及的核心概念

| 概念 | 作用 |
|------|------|
| Git | 版本控制，记录修改历史 |
| GitHub | 代码托管和协作平台 |
| Node.js | 服务器端 JavaScript 运行环境 |
| npm | 包管理器，安装 Express 等依赖 |
| Express | Web 框架，简化后端路由和请求处理 |
| REST API | 使用 HTTP 方法的增删改查规范 |
| JSON | 前后端数据交换格式 |
| 静态文件托管 | Express 直接提供 HTML/JS/CSS |
| fetch + async/await | 前端调用后端接口的现代方式 |
| fs + JSON 持久化 | 服务端数据写入文件，重启不丢失 |
| Railway | 云部署平台，连接 GitHub 自动上线 |
| Procfile | 声明应用启动命令 |
| 中间件（Middleware） | 请求拦截层，实现认证/日志等横切关注点 |
| Base64 Token | 简易身份认证方案，密码编码后作为凭证 |
| 标签分类 | 数据分组与筛选，前后端协同过滤 |
| 截止日期排序 | 服务端排序 + 前端视觉状态反馈 |

## 架构图

```
用户浏览器
    │
    ▼
Railway 公网地址 (https://git-production-xxx.up.railway.app)
    │
    ├── / (未登录 → 登录页) 或 (已登录 → 主界面)
    ├── /api/login (登录，无需认证)
    └── /api/todos/* (需 Bearer token)
            │
            ▼
    Express 服务器
        ├── authMiddleware（基础验证）
        ├── 静态文件服务 (public/)
        └── REST API
              │
              ▼
          data.json（持久化存储）
```

## API 完整列表

| 方法 | 路径 | 认证 | 参数 | 说明 |
|------|------|------|------|------|
| POST | /api/login | 否 | `{ password }` | 登录获取 token |
| GET | /api/todos | 是 | `?q=&tag=&overdue=` | 获取列表，支持搜索/筛选/过期 |
| GET | /api/todos/:id | 是 | — | 获取单个事项 |
| POST | /api/todos | 是 | `{ text, tag?, dueDate? }` | 新增事项 |
| PUT | /api/todos/:id | 是 | `{ text?, done?, tag?, dueDate? }` | 更新事项 |
| DELETE | /api/todos/:id | 是 | — | 删除事项 |
| GET | /api/tags | 是 | — | 获取所有标签 |
